# 阶段 16B：三台真实设备局域网验收

目标：在阶段 16A 合并后，使用三台真实设备验证 macOS、Linux、mDNS、VPN、后台生命周期和真实 Pi Agent 通信。

状态：等待真实设备。本阶段不能由当前 Codex 环境独立完成。

依赖：阶段 16A。

## 执行边界

- 阶段 16A 可以独立提 PR、通过 CI 并合并；本阶段不阻塞阶段 16A 合并。
- 未获得三台真实设备前，本文件保持在 `in-progress`，所有真实验收项保持未勾选。
- GitHub 托管 runner 和单机三进程不能代替本阶段。
- 未完成本阶段前，不得宣称“跨平台真实局域网验收通过”。
- 完成本阶段不自动创建 Release；发布仍需用户明确说出 `RELEASE vX.Y.Z`。

## 所需条件

- [ ] 三台真实设备连接同一个 Wi-Fi 或有线普通网络。
- [ ] A 使用 macOS，B 使用 Ubuntu/Linux，C 使用 macOS 或 Ubuntu/Linux。
- [ ] 第二轮交换 A、B 角色，分别验证 macOS 和 Linux 建群。
- [ ] B、C 中至少一台安装可开关的 VPN。
- [ ] 三台设备安装相同的 Pi 正式版，当前基线为 `0.80.10`。
- [ ] 三台设备通过同一个 commit SHA 安装 Pi Comms 待发行版本。
- [ ] 三台设备可以使用各自正常配置的真实模型和 API Key。
- [ ] 准备两个普通网络，例如普通路由器网络和手机热点。

## 安装与数据边界

- [ ] 使用 `git:github.com/LetItBe12345/pi-comms@<commit SHA>` 固定安装。
- [ ] 禁用项目 `.pi/extensions/` 和 `pi -e` 开发版。
- [ ] 报告记录 Pi、Pi Comms、Node.js、系统版本和运行代码 SHA。
- [ ] 三台设备分别生成 A、B、C 一次性仓库，彼此不共享磁盘或 Git 工作区。
- [ ] 验收报告不保存 API Key、完整邀请码、成员凭证、IP、Wi-Fi 名称或真实目录。

## 附近发现与加入

- [ ] A 创建“允许附近设备加入”的群组；界面不显示 Broker 或“协作空间”。
- [ ] B、C 在“附近群组”中看到群组名称和在线人数。
- [ ] B、C 输入该群唯一的邀请码后加入。
- [ ] 正确邀请码可以加入。
- [ ] 错误邀请码不能获得成员列表和群聊快照。
- [ ] 禁用 mDNS 后，完整群组邀请信息仍能直接加入并完成通信。
- [ ] 群组首次出现在附近列表的时间不超过 10 秒。
- [ ] 输入邀请码后完成加入的时间不超过 5 秒。

## 私有代码协作与真实 Agent

- [ ] A 仓库只保存 API 协议与需求。
- [ ] B 仓库只保存服务端实现。
- [ ] C 仓库只保存客户端或验收测试。
- [ ] 验证人对人公开消息，送达时间不超过 3 秒。
- [ ] A 向 B-Pi 发送定向代码任务。
- [ ] B-Pi 修改自己的私有仓库并公开回答。
- [ ] B-Pi 将后续任务自动转交 C-Pi。
- [ ] C-Pi 修改自己的私有仓库并公开最终结果。
- [ ] 验证忙碌任务按顺序进入队列。
- [ ] 验证最终回答只回传一次，并与原始请求正确对应。
- [ ] 报告只记录模型名称，不记录 API Key。

## 身份、成员与群主管理

- [ ] 远程成员关闭并重新打开同一 Pi Session 后，使用长期成员凭证自动回群。
- [ ] 群主关闭并重新打开同一 Pi Session 后恢复管理权限。
- [ ] fork 或 clone 群主 Session 不继承群主管理权限。
- [ ] 复制项目目录不能获得群主管理权限。
- [ ] 重新生成邀请码不影响已有成员，旧邀请码立即失效。
- [ ] 普通成员不能生成或修改邀请码。
- [ ] 群主移出成员后，该成员的长期凭证失效。
- [ ] 使用临时测试群验证解散后，三台设备上的群组都消失。
- [ ] 不测试群主转移；群主所有权与 Broker 所有权保持一致。

## 断线、VPN 与普通网络切换

- [ ] 一台设备断网后重连，并在 10 秒内恢复原身份。
- [ ] VPN 关闭、开启、再次关闭不改变群组、成员、邀请码或历史。
- [ ] VPN 允许本地网络时，附近发现和群聊在 15 秒内恢复。
- [ ] VPN 的 Kill Switch 阻止局域网时，界面明确提示 VPN 正在阻止附近连接。
- [ ] 网络 A 切换到网络 B 后保留群组数据，但未经确认不在网络 B 广播。
- [ ] 群主确认网络 B 后才恢复附近可见。
- [ ] 返回已确认的网络 A 后，原成员不需要重新输入邀请码。
- [ ] VPN 接口开关不被误判为普通网络切换。

## Broker 生命周期与发行版升级

- [ ] 关闭客户端不会意外启动第二个 Broker。
- [ ] 两台本机 Pi 并发首次进入 `/comms` 只启动一个 Broker。
- [ ] `local` 切换 `lan-host` 后不存在共享同一 SQLite 的第二个 Broker。
- [ ] 所有 Session 关闭后，默认按空闲策略暂停附近加入。
- [ ] 开启“本次开机保持可加入”后，关闭 Pi Session 仍可被附近设备发现。
- [ ] 真实重启群主设备后，临时后台设置失效。
- [ ] 开启“登录后自动开放”并再次重启后，原 Broker 和原群组在 30 秒内恢复。
- [ ] 从真实 `v0.1.0` 数据库原地升级到待发行版本。
- [ ] 升级时保持旧发行版 Broker 和至少一个 Session 正在运行。
- [ ] 新发行版优雅重启旧 Broker，并保留数据库、群组、历史和长期成员。
- [ ] 未完成请求明确标记为“群聊服务重启，任务已中止”，且不会重复执行。
- [ ] 开发版不能静默接管正在运行的发行版。

## TUI、故障提示与证据

- [ ] macOS 和 Linux 各验证一次 80×24 最低终端尺寸。
- [ ] 使用 120×36 保存推荐展示截图。
- [ ] 首屏完整显示“我的群组”“已加入”“附近群组”。
- [ ] 当前页面清楚显示可用快捷键。
- [ ] 中文群名、用户名和错误提示不出现错位或乱码。
- [ ] 防火墙、客户端隔离、无私网 IPv4 和 VPN 拦截均显示用户可理解的提示。
- [ ] 保存附近发现、三人入群、Agent 接力和断线恢复的脱敏截图。
- [ ] 保存脱敏 Markdown 报告到 `docs/acceptance/lan-YYYY-MM-DD/`。
- [ ] 报告记录首次失败、重试原因、实际耗时和最终结果。

## 稳定性标准

- [ ] mDNS 发现连续执行 3 次，全部在 10 秒内成功。
- [ ] 断线恢复连续执行 3 次，全部在 10 秒内成功。
- [ ] VPN 开关连续执行 3 次，全部在 15 秒内恢复。
- [ ] Broker 重启连续执行 3 次，全部在 15 秒内恢复。
- [ ] 产品缺陷修复后重新执行受影响的完整验收段，不只补测失败步骤。
- [ ] 操作或网络环境导致的重试必须在报告中记录原因。

## 实施与合并顺序

1. 等待阶段 16A 合并，并准备三台真实设备。
2. 使用同一个待发行 commit SHA 完成全部真实验收。
3. 若发现产品缺陷，单独修复、通过 CI，并重新执行受影响的完整验收段。
4. 提交只包含验收记录、截图、最终文档和 TODO 状态的 PR。
5. 验证最终提交与报告中的运行代码树一致。
6. CI 通过后合并验收记录 PR，并将本文件移入 `TODO/done/`。

## 本阶段不做

- 不用 GitHub 托管 runner、虚拟机或单机多进程冒充三台真实设备。
- 不绕过 VPN 软件主动启用的局域网封锁。
- 不支持 Windows、IPv6 或主机名连接。
- 不做公网服务器、云端账号、NAT 穿透、TURN 或中继。
- 不做跨子网 mDNS 代理和多个 Broker 数据复制。
- 不自动创建 Tag 或 GitHub Release。

## 完成条件

- [ ] macOS 与 Linux 可以真实发现并加入同一个附近群组。
- [ ] 三台设备完成四种通信方向、私有代码接力和唯一回答验收。
- [ ] 断线、VPN、普通网络切换、后台、自启和升级验收全部通过。
- [ ] 自动发现失败时，完整邀请信息仍可完成全部通信。
- [ ] 脱敏报告和必要截图已提交。
- [ ] 文档准确说明正式支持平台、网络要求和当前限制。
