# Pi Session 群聊通信工具 PRD
## 1. 产品目标

开发一个运行在 Pi CLI 中的群聊通信工具。

用户通过命令进入 TUI。不同用户和他们当前运行的 Pi Session 可以加入同一个群组，在群内进行公开通信。

核心能力：

> 将群聊中的定向消息注入指定 Pi Session，并把 Pi 的回答重新发送到群聊。

---

## 2. MVP 范围

第一版只支持：

* Pi Agent
* 同一台设备
* 纯文本消息
* 单个用户只携带一个 Pi Session
* 单个用户同时只加入一个群组
* 人与人通信
* 人与 Agent 通信
* Agent 与 Agent 通信
* TUI 群聊界面

第一版不支持：

* 跨设备通信
* 其他 Agent Harness
* 图片、文件和代码附件
* 群聊摘要
* 上下文压缩
* 全局可读
* Agent 持续监听所有群聊消息

---

## 3. 核心关系

每个运行中的 Pi Session 对应一个真实用户。

用户进入群聊时，会同时产生两个群聊成员：

* 真实用户
* 当前 Pi Session

例如：

```text
张三
张三的 Pi
李四
李四的 Pi
```

人和 Agent 在群聊中的表现形式一致：

* 都有独立名称
* 都可以发送公开消息
* 都可以被 `@`
* 都显示在同一个成员列表中

系统内部可以记录成员类型，但 UI 不需要将人和 Agent 分区。

---

## 4. 名称管理

用户进入群组前，需要设置：

* 用户名称
* Agent 名称

名称由用户自行定义。

规则：

* 同一个群内不能出现重复名称
* 用户名称和 Agent 名称之间也不能重复
* 名称冲突时，禁止进入群组
* 用户必须修改名称后才能继续

名称只用于群聊展示和 `@` 定位。

系统内部仍使用唯一 ID 路由消息。

---

## 5. 群组规则

一个用户同时只能加入一个群组。

如果用户希望加入其他群组，必须先退出当前群组。

群组可以通过以下方式发现或加入：

* 邀请码
* 群组 ID
* 局域网发现

MVP 优先实现同一设备上的群组发现。

后续再扩展局域网设备发现。

---

## 6. 用户入口

Pi 提供一个初始命令，例如：

```text
/comms
```

执行后进入 TUI，并完成以下流程：

1. 检查当前 Pi Session。
2. 设置用户名称。
3. 设置 Agent 名称。
4. 选择创建或加入群组。
5. 检查名称冲突。
6. 进入群聊界面。

大部分操作都在 TUI 内完成，不依赖外部 slash command。

---

## 7. TUI 界面

TUI 至少包含以下区域。

### 群组信息

显示：

* 当前群组名称
* 在线成员
* 用户和 Agent 的状态

### 消息区域

显示所有公开群聊消息：

* 发送者名称
* 消息内容
* 发送时间
* `@` 目标
* 发送失败状态

### 输入区域

支持：

* 发送普通文本
* `@` 用户
* `@` Agent

### 权限设置

Agent 主人可以在 TUI 中修改当前 Session 的接收权限。

---

## 8. 消息公开规则

系统不提供私聊。

所有消息都公开显示在群聊中。

例如：

```text
张三：@李四的 Pi 检查这个函数。
```

虽然消息只会注入 `李四的 Pi`，但所有群组成员都能看到。

Agent 回复也是公开消息：

```text
李四的 Pi：@张三 发现一个空指针问题。
```

Agent 对 Agent 的消息同样公开：

```text
张三的 Pi：@李四的 Pi 请检查测试结果。
```

这条消息会：

1. 显示在群聊中。
2. 尝试注入目标 Pi Session。

---

## 9. Agent 消息接收规则

### 9.1 默认行为

Agent 默认不会读取普通群聊消息。

只有消息明确 `@Agent` 时，系统才尝试将消息注入该 Agent Session。

适用于：

* 人 `@Agent`
* Agent `@Agent`

没有 `@Agent` 的消息只显示在群聊中。

---

### 9.2 注入内容

MVP 每次只注入当前消息。

注入内容包含：

* 消息发送者
* 当前群组名称
* 当前消息正文
* 当前消息中的 `@` 目标

不注入：

* 整个群聊历史
* 最近消息窗口
* 自动摘要
* 其他无关消息

---

## 10. Agent 权限

每个 Agent Session 的权限由它的主人控制。

权限在 TUI 中设置。

支持三种模式：

### 自动接收

默认模式。

Agent 被 `@` 后，消息立即进入 Session。

### 需要批准

Agent 被 `@` 后，消息进入待批准列表。

主人在 TUI 中批准后，消息才会进入 Session。

### 禁止接收

Agent 被 `@` 后，消息仍显示在群聊中，但不会进入 Session。

发送者会看到接收被拒绝。

---

## 11. Agent 忙碌状态

如果 Agent 正在处理任务，又收到新的有效消息：

* 新消息进入队列
* 不打断当前任务
* 按接收顺序依次处理

MVP 不支持：

* 抢占当前任务
* 调整消息优先级
* 并行处理多个群聊请求

---

## 12. Agent 输出规则

消息进入 Pi Session 后，Pi 正常执行。

Pi 本次任务的最终回答会自动发送到群聊。

例如：

```text
张三：@李四的 Pi 检查 server.py。

李四的 Pi：@张三 server.py 第 48 行缺少异常处理。
```

MVP 只发送最终回答。

不发送：

* 模型内部推理
* 工具调用细节
* 中间 Token 流
* 无关终端输出

---

## 13. Agent 对 Agent 通信

Agent 可以在回答中 `@` 另一个 Agent。

例如：

```text
张三的 Pi：@李四的 Pi 请运行测试并返回结果。
```

系统识别该消息后：

1. 将消息公开显示在群聊中。
2. 检查目标 Agent 是否在线。
3. 检查目标 Agent 的接收权限。
4. 将消息注入目标 Session。
5. 目标 Agent 的回答再次显示在群聊中。

---

## 14. 防止 Agent 循环通信

Agent 对 Agent 的连续自动通信最多允许 10 轮。

达到上限后：

* 停止继续注入消息
* 在群聊中显示已达到轮数限制
* 等待真实用户决定是否继续

一轮定义为：

> 一个 Agent 向另一个 Agent 发送消息，并获得一次回答。

---

## 15. 在线与离线规则

用户进入 TUI 并加入群组后：

* 用户在线
* Agent 在线
* 可以接收群聊消息

退出 TUI 后：

* 用户退出群组
* Agent 退出群组
* 群聊连接断开
* Pi 进程可以继续在后台运行
* 但不再接收群聊消息

Pi Session 关闭后：

* 用户和 Agent 都从群组成员列表中消失
* 不保留离线成员状态

离线成员不能被成功 `@`。

系统只显示发送失败，不保存离线任务。

---

## 16. 数据保存

MVP 保存：

* 群组信息
* 在线成员信息
* 群聊文本消息
* 消息发送时间
* 消息发送者
* `@` 目标
* 消息发送状态
* Agent 通信轮数

MVP 不保存：

* Agent 私有上下文
* Pi Session 完整上下文
* 模型推理过程
* 终端完整输出

---

## 17. 系统模块

### Pi Extension

负责：

* 注册 `/comms` 命令
* 获取当前 Pi Session
* 启动 TUI
* 将消息注入 Pi
* 获取 Pi 最终回答

### TUI Client

负责：

* 用户和 Agent 命名
* 群组创建与加入
* 消息展示和输入
* Agent 权限设置
* 待批准消息处理
* 在线成员展示

### Message Router

负责：

* 群聊消息广播
* `@` 目标解析
* 在线状态检查
* 消息队列
* Agent 消息注入
* 通信轮数限制

### Local Message Store

负责保存群聊历史和必要状态。

---

## 18. MVP 完整流程

1. 用户 A 启动 Pi Session。
2. 用户 A 输入 `/comms`。
3. 用户 A 设置用户名称和 Agent 名称。
4. 用户 A 创建群组。
5. 用户 B 在另一个 Pi Session 中进入 `/comms`。
6. 用户 B 加入同一个群组。
7. 群内出现四个成员：

```text
用户 A
用户 A 的 Pi
用户 B
用户 B 的 Pi
```

8. 用户 A 和用户 B 可以公开聊天。
9. 用户 A `@用户 B 的 Pi`。
10. 消息进入目标 Pi Session。
11. 目标 Pi 执行任务。
12. Pi 的最终回答发送到群聊。
13. Pi 可以继续 `@` 另一个 Pi。
14. Agent 间最多自动通信 10 轮。
15. 用户退出 TUI 后，其用户和 Agent 同时离开群组。

---

## 19. Roadmap

### 全局可读模式

Agent 主人可以允许 Session 接收群内所有消息，而不要求显式 `@`。

该功能暂不进入 MVP。

原因：

* 需要解决消息进入上下文但不触发回答的问题
* 容易快速消耗上下文
* 不同 Agent Harness 的支持方式不同

### 上下文管理

未来可以增加：

* 最近消息滑动窗口
* 群聊滚动摘要
* 按任务检索相关消息
* Agent 自主判断是否响应

这些方案可能需要额外模型调用和 API 成本，MVP 暂不实现。

### 跨设备通信

未来支持：

* 局域网发现
* 邀请码跨设备加入
* 中继服务器
* 远程 Session 通信

### 支持更多 Agent

完成 Pi 版本验证后，再开发：

* Oh My Pi
* OpenClaw
* Hermes Agent
* 其他 Agent Harness

---

## 20. 产品定义

该产品不是多 Agent 编排框架，也不是普通聊天室。

它是：

> 一个将真实用户和当前运行的 Pi Session 放入同一公开群聊，并通过 `@` 将文本消息安全注入指定 Session 的终端通信工具。
